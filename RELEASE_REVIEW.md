# Kaufman Finance CRM — Enterprise Release-Readiness Review
_Generated by a 21-role multi-agent review board (152 agents, ~7M tokens). Live Vercel app + new Supabase Frankfurt DB._

**Verdict: Not Ready | Score: 38/100** — 126 findings (8×P0, 43×P1, 49×P2, 26×P3). 37/40 P0/P1 survived adversarial verification.

## 1. Executive Summary
VERDICT: Not Ready | SCORE: 38/100

## Executive Assessment

This is an unusually well-engineered codebase — disciplined layered architecture, defense-in-depth RLS, encrypted tokens/backups, audit partitioning, optimistic locking, and a real pgTAP test suite. The *code* is largely production-grade. **What makes it Not Ready is not code quality — it is the operational, legal, and data-integrity envelope around a LIVE app holding Israeli PPL-regulated financial PII.**

Three categories block release. **(1) Security/Privacy:** a confirmed cross-case IDOR — `task_comments` RLS is `USING (deleted_at IS NULL)` with zero case scoping, so any authenticated advisor can read every task thread office-wide via the anon key; MFA is half-built and unenforced (password-only ceiling); and there is no right-to-erasure path for borrower PII (national IDs persist forever after case purge). **(2) Backup/Recovery:** the nightly backup currently silently no-ops (Drive unconnected → HTTP 200 skip), the staleness watchdog is not even scheduled in `vercel.json`, and restore is merge-only (cannot repair corruption). For ~80 real PII cases there is effectively no working application backup and no alarm. **(3) Legal/Release:** no in-app privacy/terms/consent surface and no signed DPA for a regulated product; retention defaults hard-purge client records 14 days after soft-delete (self-documented as illegally short); and the live Vercel deploy path has NO migration gate — the `convert_lead_to_case` bug that was runtime-broken across 70 migrations proves prod silently diverges. Operator runbooks describe the *wrong runtime entirely* (Vultr Docker, not Vercel). Compounding everything, Sentry is off, so all of these fail invisibly.

These are go-live gating defects, not polish. The path to "Conditionally Ready" is concrete and mostly small-diff (one RLS policy, one `vercel.json` line, retention config, legal wiring), but until they land this should not hold real client data.

## Top 20 Highest-Risk Findings

1. **[P0] task_comments cross-case IDOR** — `USING (deleted_at IS NULL)`; any authed user reads every task thread via PostgREST. Fix RLS to gate on `can_view_case`. (migration 120:52-54)
2. **[P0] Nightly backup silently no-ops + watchdog unscheduled** — Drive disconnected → 200 skip; `backup-watchdog` absent from `vercel.json`. No working backup, no alarm, for 80 PII cases.
3. **[P0] No in-app legal/consent + no lawful basis** — no /privacy, /terms, consent capture, or recorded legal basis for PPL-regulated PII intake.
4. **[P0] Legal docs are unreviewed drafts (69 [[FILL]] placeholders)** — never counsel-reviewed; not surfaced anywhere live.
5. **[P0] No DPA** between developer-operator (processor) and office (controller).
6. **[P0] No migration-before-deploy gate on live Vercel path** — `deploy.yml` disabled; manual psql; `convert_lead_to_case` was runtime-broken 053→123. Prod can silently diverge.
7. **[P1] No right-to-erasure for borrowers** — orphaned national IDs/financial PII persist forever after case purge; no `permanently_delete_borrower`.
8. **[P1] Retention default 14-day hard purge** — erases client mortgage records 14 days after soft-delete; self-documented as illegally short vs. multi-year duties.
9. **[P1] Scheduled-job execution unverifiable** — Vercel crons need `CRON_SECRET`; pg_cron needs dashboard toggle; both silent if missing → no backups/erasure/audit-purge.
10. **[P1] PII-file erasure depends on a possibly-dead cron** — if cleanup cron is down, files orphan and the 30-day backstop deletes the only pointer → un-erasable PII (PPL breach).
11. **[P1] MFA dormant/unenforced** — TOTP built but login/middleware never demand AAL2; password-only ceiling for national-ID/bank data.
12. **[P1] Sentry off in prod** — zero error visibility; every silent failure above is invisible; source maps unwired.
13. **[P1] Operator runbooks describe wrong runtime** — DEPLOYING/RESTORE point to dead Vultr Docker host; incident response would target the wrong system.
14. **[P1] No DSAR tooling** — only bulk all-cases export; no per-subject access or targeted erasure.
15. **[P1] Individual PII-document access & bank-PDF generation not audited** — "who viewed this client's ID" is unanswerable.
16. **[P1] Returning-client add overwrites shared borrower fields** — cross-case clobber of another advisor's borrower data (mig 123 reuse path).
17. **[P1] Sub-processor register factually wrong** — lists Vultr not Vercel; unresolved region placeholders; undermines DPA/privacy compliance.
18. **[P2] Password reset fails silently** — uses Supabase SMTP (likely unconfigured), returns `sent:true` regardless; locked-out manager has no recovery.
19. **[P2] XLSX export formula/CSV injection** — borrower-controlled name/note written raw; live formula on manager's machine → exfil/RCE on legacy Excel.
20. **[P2] Whole application layer (~90 server actions) + cron routes have zero tests** — the auth/IDOR/error-shaping glue most likely to regress is uncovered; RLS suite only runs on migration-touching PRs.

### Product Management
**Readiness:** Conditionally Ready

**Strengths**
- The core lead->case->execution loop is complete and coherent: convert-lead (convert-lead.ts:54) redirects straight to /cases/[id]/edit where the advisor-assignment field lives (case-form.tsx:127-128), and every case sub-route (edit, borrowers, banks, documents, simulators, history) exists with no 404 dead-ends.

- Document-request and client-message flows are dual-channel (WhatsApp deep-link + email) with the WhatsApp text auto-listing the still-missing required checklist items (send-doc-request-button.tsx:184-191) — a thoughtful, friction-reducing touch for a non-technical office.

**Weaknesses**
- The Templates feature is fully shipped (admin-visible /settings/templates tab, fully translated, CRUD working) but has ZERO consumers — both message-send paths build bodies from hardcoded i18n strings, so authored templates are unreachable. The UI subtitle even promises 'variables replaced at send time' that does not exist.
- There is no way to find unassigned cases: convert leaves assigned_advisor_id NULL (mig 124:91) and the dashboard advisor filter only matches a specific advisor ID (case-filters.ts:66) — no 'unassigned' option or badge.
- No legal/consent surface inside the live app (Terms / Privacy / consent checkbox) despite handling Israeli-PPL-regulated financial PII; docs/legal/*.md exist on disk but are not rendered or linked anywhere users can see.
- The email document-request (send-document-request.ts) depends on Resend, which is a pending ops item; if unconfigured it returns 'not_configured' and the email silently never arrives.
- Status changes are entirely free-form (any stage to any stage) with no transition guidance — acceptable flexibility, but offers the non-technical office no guard-rails or 'next step' nudges across the 12-stage pipeline.

**Risks**
- A non-technical manager will spend time authoring WhatsApp/email templates (the UI invites it) and reasonably expect them when sending messages — then find them missing with no error, eroding trust in the product on day one.
- A converted lead that the manager doesn't assign at the edit screen becomes hard to rediscover later (no unassigned filter/badge), risking a client silently falling through the cracks — the exact failure a CRM exists to prevent.
- Absent in-app consent/privacy text is a regulatory exposure for a live product processing national IDs and financial data under Israeli PPL.
- If Resend stays unconfigured, the email half of every client-communication flow is a silent no-op, pushing the office back to manual email and undercutting adoption.

**Immediate Priorities**
- Decide Templates' fate before onboarding Kaufman: either wire a template-picker into send-client-message-button / send-doc-request-button (and the email body in send-document-request.ts), or hide the /settings/templates tab so it doesn't promise unbuilt behavior.

- Surface Privacy/Terms (and a first-login consent acknowledgement) inside the app, linking the docs/legal/* content.
- Confirm Resend is configured in prod so the email document-request and invite emails actually send (currently a silent-fail dead-end if unset).

## (1) Should this be released today?

**No. Do not release to real client data today.** It is already live, which makes this worse, not better — the app is in production processing (or about to process) ~80 real mortgage cases of Israeli PPL-regulated financial PII while carrying **multiple P0 defects across security, privacy, legal, and disaster-recovery simultaneously**.

The codebase engineering quality is genuinely high (clean layering, RLS depth, audit triggers, encryption at rest). That is exactly the trap: the *code* is good enough to feel ready, while the things that actually get a finance office sued or wiped out — a cross-case data leak, no working backup, no legal basis to process PII, and a deploy path that can silently 500 — are open. **Engineering polish is not release readiness.**

If real cases have not yet been imported: **freeze the import** until the blockers below clear. If they have, treat this as an active incident, not a launch.

## (2) Hard release blockers (must-fix-first)

These block go-live unconditionally.

**Security / data-integrity**
- **B1 — `task_comments` cross-case IDOR** (SEC-idor-task-comments / BE-task-comments-idor, mig `120:52-54`). RLS is `USING (deleted_at IS NULL)` — *any* authenticated user reads *every* task's thread via the anon key over PostgREST, including private tasks and cases they cannot see. This is a live confidentiality breach and arguably a PPL breach. The "enforced at app layer" comment is false — the action only checks `getUser()`.
- **B2 — No migration-before-deploy gate on the live Vercel path** (deploy.yml disabled; manual `psql` per memory). `convert_lead_to_case` was runtime-broken from mig 053→123 referencing a non-existent column and nobody's CI caught it. With parallel agents committing migrations, one missed manual apply = 500s on a feature. Prod has **no authoritative record of which of 001..139 are applied.**

**Backup / recovery (DR)**
- **B3 — There is effectively no working backup.** Nightly backup returns HTTP 200 `skipped:'drive_not_connected'` when Drive is unconnected (current state), and the **backup-watchdog is not even scheduled in `vercel.json`** — so a silently-absent backup raises zero alarm. For 80 PII cases the only net is Supabase PITR, itself an unverified manual Pro toggle.
- **B4 — Scheduled jobs are unproven.** All 4 Vercel crons run only if `CRON_SECRET` is set (optional in env.ts → silent 503s); all 4 pg_cron jobs run only if `pg_cron` was enabled in the dashboard (`CREATE EXTENSION` is a no-op otherwise). Neither is verifiable from code; both are unchecked go-live steps.

**Legal / privacy (regulatory)**
- **B5 — No lawful basis + no consent capture + no in-app privacy/terms** (LEGAL-1/3, PRIV-consent-notice). Live product processes national IDs/incomes/bank data with the lawful-basis field still `[[FILL]]` and no surface where a data subject is informed or consents.
- **B6 — No DPA** between operator (processor) and office (controller) (LEGAL-2). Referenced in Terms, file does not exist.
- **B7 — Retention default of 14-day hard purge** (`office_settings.deleted_records_retention_days=14`, mig 010). The project's own legal draft says this is "almost certainly too short." pg_cron will hard-erase mortgage records 14 days after soft-delete — directly conflicting with multi-year financial record-keeping duties. Must be raised on the **prod** row before import.
- **B8 — No right-to-erasure for borrowers** (PRIV-erasure-borrower). Orphaned borrowers retain full PII forever; no `permanently_delete_borrower`, nothing sets `deleted_at` on orphans. A PPL erasure request cannot be honored.

**Auth**
- **B9 — MFA is half-built and unenforced** (SEC-mfa-dormant). Single-factor password-only is the real ceiling for an app holding financial PII; the `set-password` path also accepts weaker passwords than the change-password form. Either wire AAL2 or formally sign off password-only as accepted risk — do not ship the half-rendered state.

## (3) Hidden risks (look fine, aren't)

- **Stale ops docs point at the wrong runtime.** DEPLOYING.md / RESTORE_RUNBOOK / INCIDENT_RESPONSE all describe the dead **Vultr Docker** host and say "Vercel cron does not fire." During an incident the operator SSHes to an irrelevant box and wastes recovery time. The bootstrap step in KAUFMAN_GO_LIVE.md is also **wrong** (raw `UPDATE profiles` affects 0 rows; correct path is `bootstrap_first_admin` RPC) — the manager could be unable to log in on day one.
- **Returning-client overwrite** (BE-returning-borrower-cross-case): adding a returning borrower blind-overwrites the shared borrower entity's name/ID/phone with the new typed values, corrupting another advisor's case. Mig 064 documents this exact hole; it was not actually closed.
- **Restore is merge-only.** "Restore" undeletes missing rows but **cannot repair a corrupted/maliciously-edited row** — false safety in a data-corruption incident.
- **XLSX export formula injection** — borrower-named `=HYPERLINK(...)` executes when the manager opens the export.
- **Statistics will be born wrong.** Import stamps execution/creation dates as import-day and can't import `requested_mortgage_amount` → "active loan volume" reads ~0 and the trend shows a fake spike. The manager's first impression of the dashboard is garbage data.
- **Sentry is off** — no production error visibility, so every silent failure above is invisible until restore/erasure day.
- **`task_comments` and `set_primary_bank` on soft-deleted rows, TOCTOU on last-borrower** — latent integrity bugs with no test coverage (entire ~90-action application layer is untested; RLS tests run only on migration-path PRs).
